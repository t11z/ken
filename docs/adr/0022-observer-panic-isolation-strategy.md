# ADR-0022: Observer panic isolation strategy

- **Status:** Accepted
- **Date:** 2026-04-11
- **Deciders:** —
- **Supersedes:** —
- **Superseded by:** —

## Context

ADR-0018 committed Ken's observer subsystem to a specific failure-isolation property: *"A panic inside an observer is caught at the `spawn_blocking` boundary and converted into a tick-level failure for that observer alone. The observer's last known value (if any) is used for the snapshot. Other observers and the worker loop are unaffected. An observer that panics repeatedly is logged but not disabled — disabling is a behavior change that would require its own decision."* The intent is plain: a panicking observer is a transient bug, not a terminal one, and the system should keep retrying it on every tick so that the next successful collection lands as soon as the underlying cause goes away.

The Phase 1 closure audit found that the implementation does not match this intent. In `crates/ken-agent/src/observer/snapshot.rs`, the worker loop's per-observer slot is wrapped in an `Option<Box<dyn Observer>>`. When `spawn_blocking` returns a `JoinError` indicating a panic, the slot is set to `None` and the observer is never reconstructed. From the next tick onward, the observer is silently absent: the loop sees `None`, falls through to the cached value (or to `Unobserved` if there is none), and never invokes the observer again until the agent process is restarted. The audit named this "disable on first panic," and that label is accurate. The code's behavior is the opposite of what ADR-0018 specifies, even though the surrounding mechanics — the catch boundary, the cached fallback, the panic log — all look correct in isolation.

The drift was not introduced deliberately. It emerged because the Rust mechanics make it the path of least resistance: a `tokio::task::spawn_blocking` call moves its closure (including the observer) into the spawned task, and the closure is not returned to the caller on panic. To preserve the observer across a panic, the implementation has to either keep the observer outside the closure (and clone/borrow into it), wrap the panic-prone work in `std::panic::catch_unwind` *inside* `spawn_blocking` so the observer struct never leaves the closure under unwinding, or use a different ownership shape entirely. None of these is hard, but none is the obvious first thing an implementer reaches for, and ADR-0018 did not pin down which to use because at the time the trait shape was deliberately left to the implementing prompt.

The decision is forced now because the Phase 1 closure audit identified the gap as blocking, and the same observer refactor in ADR-0021 has just landed, which means the surrounding code is in flux and the right moment to settle the question is before it solidifies in a different direction. A decision here also unblocks the audit aufschreibung: without it, the `phase-1-closure-audit.md` document would have to record an unresolved drift against ADR-0018, which is exactly the kind of repository state Ken's ADR discipline exists to prevent.

This ADR governs *how* observer panic isolation is implemented, given that ADR-0018's stated intent is treated as binding. It does not relitigate whether "logged but not disabled" is the right strategy — that question is reopened only briefly in the Alternatives section, with the explicit conclusion that the alternatives are Phase 2 material.

## Decision

Observer panic isolation is implemented by **catching unwinds inside the `spawn_blocking` closure** rather than at the `JoinError` boundary outside it. The worker loop owns each observer as an `Arc<Mutex<O>>` (or an equivalent shape that allows the observer to be borrowed into the spawned closure without being moved out of the slot). When the closure runs, it acquires the observer via the `Mutex`, performs the observation under `std::panic::catch_unwind`, and produces either `Ok(value)` or `Err(panic_info)`. The observer struct itself is never moved into the closure and is never lost on unwinding.

The contract has the following properties, which together define what an implementer must satisfy and what future readers can rely on.

**Observer lifetime is independent of panic frequency.** An observer that panics on tick N is the same observer instance on tick N+1. Its internal state — last known value, refresh policy, any per-observer fields it carries — is preserved across panics. The only thing the panic produces is a logged event and a tick-level fallback to the cached value (or `Unobserved` if no cache exists yet).

**Catching happens inside the closure, not outside.** The implementation does not rely on `JoinError::is_panic()` to detect panics, and it does not use the `Option::take`-then-`Option::insert` pattern to "rescue" the observer. Both are subtler than they look: the first depends on Tokio internals the agent should not couple to, and the second has a window during which the slot is `None` and a concurrent reader would see a missing observer. The `catch_unwind` approach is local to the closure and has no such window.

**`catch_unwind` requires `UnwindSafe`, and the trait reflects that.** The `Observer` trait gains an `UnwindSafe` bound (or the observer is wrapped in `AssertUnwindSafe` at the call site, with a doc comment explaining the choice — but the `UnwindSafe` bound is preferred because it forces implementers to think about it). Observers that hold types which are not `UnwindSafe` by default (anything with interior mutability across the unwind boundary, e.g. `RefCell`, `Cell`, or unprotected raw pointers) must demonstrate that their state is consistent across an unwind, or wrap the unsafe parts in their own `AssertUnwindSafe` with a justification. This is mechanical work, not architectural work, but it has to be uniform across all five observers.

**Logging is rate-limited at the source.** A panicking observer that fires once per tick will produce a log entry once per tick. Without rate limiting, an observer that panics deterministically on every call generates a log entry every few seconds for the lifetime of the process, which is a real operational concern at deployment scale (even at family-IT scale). The implementation rate-limits the log emission per observer to at most one entry per ten minutes after the third consecutive panic, with the exact cadence configurable via a constant in the worker loop module. The first three panics are logged unconditionally so that a transient bug is visible immediately. After that, the logger emits a single rate-limited entry naming the observer, the panic message, and the elapsed time since the previous suppressed log. This is the only place rate-limiting is applied; observers themselves do not rate-limit anything.

**Failure isolation between observers is preserved exactly.** A panic in observer X has zero effect on observer Y, on the worker loop, on the heartbeat schedule, or on any state outside observer X's own slot. This was true in the previous (broken) implementation as well, and it remains true; what changes is only that observer X is itself preserved across the panic.

**The fallback value semantics from ADR-0019 are unchanged.** When the observer panics, the loop produces a snapshot value drawn from the observer's existing cache, with whatever `Fresh`/`Cached`/`Unobserved` tag the cache currently holds, re-tagged according to the tick boundary per ADR-0021. The panic does not introduce a new variant or a new tag; it produces the same snapshot value the loop would have produced if the observer had returned its cached value directly. From the wire format's perspective, a panic is indistinguishable from a cached read.

**No persistent disabling, no automatic recovery counters, no circuit breaker.** The strategy in this ADR is "retry every tick, log with rate limiting, never give up." It is deliberately the simplest strategy that satisfies ADR-0018's stated intent. Counted backoff and circuit-breaker patterns are explicitly out of scope and are named in Alternatives so that a future Phase 2 reopening of this question has a starting point.

This ADR governs the *strategy*. It does not prescribe a specific Rust type for the observer slot, a specific log macro, or a specific rate-limiting library. Those are implementation details for the prompt that will translate this ADR into code. The properties above are the contract; the names are not.

## Consequences

**Easier:**

- ADR-0018's stated intent and the code's actual behavior are aligned. The audit's blocking finding closes without ambiguity, and a future maintainer reading ADR-0018 sees behavior that matches the description.
- Transient observer panics self-heal. An observer that fails because of a temporary WMI lock, a momentary COM hiccup, or an unexpected response from a Windows API recovers as soon as the underlying condition clears. The family IT chief does not have to restart the service to get the subsystem back.
- The strategy is testable. A test can construct an observer that panics on its first invocation and succeeds on its second, and assert that the second invocation actually happens and produces a usable value. Under the previous implementation, this test would fail; under this one, it passes.
- Log volume is bounded by the rate limiter. A pathological observer that panics on every tick produces at most a handful of log entries per hour, not one every few seconds.

**Harder:**

- The observer slot's ownership shape is more complex. `Arc<Mutex<O>>` (or whatever the implementation chooses) is heavier than the previous `Option<Box<dyn Observer>>`, both syntactically and at runtime. The mutex acquisition is uncontended in the steady state — only one task per observer is in flight at a time — but the lock primitive exists where there was none before.
- Every observer must satisfy `UnwindSafe`, or document why it does not and use `AssertUnwindSafe`. This is a one-time cost during the refactor that implements this ADR, and a small ongoing cost when new observers are written. The doc comment on the trait must explain it clearly enough that a new observer's author does not have to reverse-engineer the constraint.
- Rate-limited logging is itself a small subsystem. The implementation has to choose a rate-limiter (a tiny per-observer counter and timestamp is sufficient — no external crate needed), and the choice has to be documented so a future reader does not assume the absence of recent log entries means the absence of recent panics.

**Accepted:**

- A deterministically-panicking observer is functionally absent until the underlying bug is fixed and a new agent version is deployed. The cached value (if any) is served indefinitely as `Cached` or `Unobserved`, and the family IT chief sees a stale subsystem in the dashboard. This is the cost of the self-healing-by-default strategy: the system trusts that panics are usually transient and accepts permanent silence as the failure mode for the rare case where they are not. The mitigation is the audit log entries, which are rate-limited but never absent — a maintainer who looks at the log within ten minutes of the next panic sees that the observer is still failing, even if the dashboard shows a comfortable stale value.
- The dashboard cannot distinguish "this subsystem is broken" from "this subsystem is fine but slow to refresh." Both look like `Cached` or `Unobserved`. Adding a "this observer is currently in a panic loop" indicator would require extending the wire format and the rendering layer; it is a Phase 2 question and is not introduced here.
- The rate-limiting threshold (three immediate logs, then one per ten minutes) is a guess at the right ergonomics for a single-tenant family-IT deployment. It will likely need tuning when the first real Windows-API observer ships and produces real panic data. The threshold lives in a constant rather than a configuration field on purpose, to avoid prematurely externalizing a tuning knob nobody has data for yet. ADR-0018's "the first deployment will likely tune them" applies here.

## Alternatives considered

**Keep the current behavior (disable on first panic) and supersede ADR-0018 in this point.** Rejected after the user-experience analysis that prompted this ADR. The argument for disable-on-first-panic was "a panicked observer might have corrupt internal state and should not run again." On closer inspection, the observer state in question is small and well-defined (last value, timestamp, refresh policy), and the typical panic source for sync FFI against Windows APIs is unexpected response shape or a NULL pointer where one was not expected — not state corruption inside the observer struct. Disabling permanently in response to a transient FFI hiccup converts a recoverable problem into a permanent one and requires manual intervention (service restart) that the family IT chief is not positioned to provide. The self-healing default is the better fit for the deployment context, and ADR-0018 was right to specify it.

**Use a circuit-breaker pattern with counted backoff.** Rejected for Phase 1 as overengineering, but explicitly preserved as the right Phase 2 answer if either of two things turns out to be true: (a) the rate-limited log volume from this ADR is still too high in practice, or (b) a permanently-broken observer becomes a recurring operational concern in a way the dashboard cannot surface. The circuit-breaker would count panics in a sliding window, disable the observer for a cooldown period after a threshold, then re-enable and try again. It is the correct upgrade path *from* this ADR, not a competing alternative *to* it. Adopting it now would mean making two decisions (default behavior + recovery strategy) at the same time without empirical data for either. Adopting it later means the data exists, the constants can be chosen, and the upgrade is mechanical.

**Catch the panic at the `JoinError` boundary outside `spawn_blocking` and reconstruct the observer.** Rejected because reconstruction requires either a constructor that the worker loop can call without arguments (which couples the loop to every observer's construction shape) or a stored "observer factory" alongside each observer slot (which doubles the slot's surface). Both are heavier than catching the panic inside the closure, and neither has any compensating advantage. The `catch_unwind`-inside approach is local to the place where the panic happens and does not propagate complexity outward.

**Wrap each observer in a separate worker thread that lives across panics, with the worker loop sending tick requests via channels.** Rejected as a re-litigation of ADR-0018's central decision. ADR-0018 already considered and rejected the multi-thread / channel-based shape, and this ADR does not reopen that choice. The observer execution model stays as ADR-0018 (and ADR-0021) define it; only the panic-handling implementation inside that model is what this ADR settles.

**Make the panic-handling strategy configurable per observer, so that some observers can be disabled on panic and others can keep retrying.** Rejected as premature optimization for a problem nobody has yet. ADR-0018's commitment to a single failure-isolation property is uniform across observers, and uniformity is itself valuable: a future reader who sees how one observer behaves can extrapolate to all of them. Per-observer configuration would mean five different policies that have to be documented, tested, and explained. If a future observer turns out to need a different policy, the right response is a follow-up ADR that names the observer and the reason, not a configuration knob that lets implementers make the choice silently.
