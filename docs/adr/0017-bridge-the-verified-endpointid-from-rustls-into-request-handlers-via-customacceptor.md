# ADR-0017: Bridge the verified EndpointId from rustls into request handlers via a CustomAcceptor

- **Status:** Accepted
- **Date:** 2026-04-10
- **Deciders:** @t11z
- **Supersedes:** —
- **Superseded by:** —

## Context

ADR-0008 establishes that the Ken agent listener uses `axum-server` with `axum_server::tls_rustls::RustlsConfig` and a custom `ClientCertVerifier` that runs the chain check, parses the leaf certificate's CN as an `EndpointId`, and verifies enrollment and revocation status against the database. ADR-0016 establishes that the `EndpointId` of an agent request derives exclusively from the verified client certificate, never from the request body. Together they require that handlers in the agent listener obtain the endpoint identity from the verified certificate, and from nowhere else.

ADR-0008 sketches the propagation mechanism in a single sentence: *"The middleware reads the peer certificate from the connection extensions (axum-server makes the rustls peer info available) and inserts the parsed `EndpointId` into the request extensions."* That sentence assumes a fact that turns out not to hold. `axum-server` 0.8 does not expose peer certificates in any form that a tower middleware can read out of an `Extension<...>` after the handshake. The upstream feature requests for this — `axum-server` issue #82 and `axum` issue #2697 — have been open for years and are not implemented. The current code in `crates/ken-server/src/http/endpoint_id.rs` documents this gap as a hard limitation and falls back to trusting the body field, which is exactly what ADR-0016 forbids.

The gap is not, however, a reason to abandon the architecture. `axum-server` exposes a public extension point, `CustomAcceptor`, designed to wrap a TLS acceptor and intervene in the connection lifecycle. Used correctly, it lets Ken extract the peer certificate from the completed `tokio_rustls::server::TlsStream` and attach the verified `EndpointId` to the per-connection state that the request layer can then read into request extensions. The feature is supported by the crate, documented in its examples, and used in the wild for exactly this kind of mTLS-with-application-identity setup. What is missing in Ken is the integration code, and what is missing in the ADR set is a record of which integration approach Ken commits to.

The decision is forced now because issues #8, #13, and #14 cannot be implemented without it. Issue #14 ("Implement Extension<EndpointId> middleware") in particular is unanswerable until this ADR exists: any prompt that tells Claude Code to "wire the verifier output into request extensions" implicitly chooses the bridge mechanism, and that choice belongs in an ADR rather than in a prompt.

## Decision

Ken implements the verifier-to-handler bridge for the agent listener by **wrapping `axum_server::tls_rustls::RustlsAcceptor` in a custom acceptor type that extracts the verified peer certificate from the completed TLS stream and attaches the parsed `EndpointId` as a connection-level extension**. A small tower middleware on the agent router then promotes that connection-level value into the per-request extensions, where handlers read it via `axum::extract::Extension<EndpointId>`.

Concretely, the design has three pieces. Their precise APIs are implementation details and may evolve; what this ADR fixes is the *shape* of the solution and the *layer* at which each concern lives.

The first piece is a `KenAcceptor` type in `crates/ken-server/src/http/tls.rs` that implements `axum_server::accept::Accept`. It owns a `RustlsAcceptor` internally and delegates the actual TLS handshake to it. After the inner acceptor completes the handshake successfully, `KenAcceptor` reaches into the resulting `tokio_rustls::server::TlsStream`, calls `get_ref().1.peer_certificates()` to obtain the chain that rustls already verified during the handshake, parses the leaf's CN into an `EndpointId` using the same logic as the existing `extract_cn` helper, and attaches the result to the connection. The attachment mechanism is whatever `axum-server`'s `Accept` trait makes available for per-connection state propagation into the request layer; the current shape is to wrap the returned IO stream and the returned service in a way that lets a downstream `tower` layer read the value. Implementation may use `http::Extensions` on the connection, a `tokio::sync::OnceCell`, or a typed wrapper struct — the choice belongs to the implementation, not to this ADR, but the *contract* is that by the time any HTTP request runs on the connection, the verified `EndpointId` is reachable from the request handling layer without going through the request body.

The second piece is a tower middleware in `crates/ken-server/src/http/endpoint_id.rs` (replacing the current placeholder file) that runs on the agent router only. The middleware's single job is to read the connection-level `EndpointId` placed there by `KenAcceptor` and copy it into `Request::extensions_mut()` so handlers can use the standard `Extension<EndpointId>` extractor. The middleware contains no parsing logic and no fallback path: if the connection-level value is absent, the middleware returns `500 Internal Server Error` and logs the condition as a server bug, because a request reaching the agent listener without a verified `EndpointId` indicates that the acceptor and the verifier have become unsynchronized — which is unrecoverable at request time and must surface loudly rather than silently.

The third piece is the wiring in `crates/ken-server/src/main.rs`: the agent listener is constructed with `axum_server::bind(addr).acceptor(KenAcceptor::new(rustls_acceptor))` instead of `axum_server::bind_rustls(addr, rustls_config)`. The admin listener is unaffected and continues to use `bind_rustls` directly with a server-cert-only configuration; ADR-0004's split between the two listeners is preserved exactly.

The verifier in `tls.rs` is unchanged. It still does the chain verification, the database lookup for enrollment and revocation, and the expiry check. The custom acceptor performs no security checks of its own — it only extracts a value that the verifier has already approved. Any failure mode in the verifier still results in a rejected handshake before the acceptor's post-handshake code runs, so the acceptor never sees an unverified certificate and cannot be tricked into attaching an unauthenticated identity.

## Consequences

**Easier:**
- The trust path becomes complete and end-to-end verifiable. Verifier → acceptor → middleware → handler is a single chain with one source of truth at each step. ADR-0016's promise — that handlers know who is talking to them, and that the body cannot lie about it — becomes operational rather than aspirational.
- The verifier in `tls.rs` stays focused on its single job (deciding whether a handshake is acceptable). The new acceptor stays focused on its single job (extracting the already-approved identity). Neither has to reach into the other's concerns.
- Issue #14 becomes implementable as a bounded, well-scoped piece of work against this ADR. Issue #13 (removing `endpoint_id` from the wire) becomes safe to land afterwards because the handlers will have a working alternative source. Issue #8 closes when both children close.
- The admin listener is structurally untouched. The acceptor change is isolated to the agent listener and cannot accidentally affect the admin path, which simplifies reasoning and reduces the surface area of the change.

**Harder:**
- Ken now owns code that touches `axum-server`'s `Accept` trait, which is a less-traveled corner of the crate's public API than `bind_rustls`. The shape of `Accept` and of `tokio_rustls::server::TlsStream::get_ref().1.peer_certificates()` has been stable across recent versions, but it is a shape Ken now depends on. A future `axum-server` major version that reorganizes the acceptor API would require Ken to follow.
- The acceptor code is the kind of code that a casual reader will find non-obvious. It must be commented heavily — not because it is clever, but because it is the seam between two libraries' lifecycle models, and seams are where confusion accumulates. The crate-level skill in `.claude/skills/mtls-with-rustls/SKILL.md` should be updated to point at this seam and explain why the verifier and the acceptor are two separate pieces.
- Tests that exercise the full mTLS round-trip become slightly more demanding. The existing integration tests in `crates/ken-server/tests/http_api.rs` reconstruct the agent router with a hand-rolled handler and never go through the real acceptor — that pattern is now actively harmful, because it would mask the bridge entirely. New end-to-end tests must use the real `KenAcceptor` against a `tokio` test server with a test CA, which is several dozen lines of setup. The integration-test rewrite is worth a separate issue and is not in the scope of this ADR, but it is a downstream consequence to plan for.

**Accepted:**
- Ken commits to maintaining a custom `Accept` implementation against `axum-server`. If `axum-server` becomes unmaintained or its `Accept` trait is restructured incompatibly, Ken absorbs the cost of migrating — either to a newer `axum-server`, to `tokio-rustls` directly with a custom acceptor loop, or to a different acceptor crate. This is the same dependency risk ADR-0008 already accepted for `axum-server`, made slightly more concrete.
- The acceptor's contract — "by the time a request runs, `EndpointId` is reachable on the connection" — is enforced by code review and tests, not by the type system. A future change that bypasses the acceptor (for example, by introducing a second listener that uses `bind_rustls` directly) would silently violate the contract for any handlers it serves. The mitigation is the middleware's hard-fail behavior on absence: a violation produces a 500 immediately rather than a silent fallback, so the regression is loud.
- There is no `axum-server`-version pin tied to this ADR. Ken's `Cargo.toml` already pins `axum-server = "0.8"`. This ADR does not introduce a new pin but does mean that any future bump must verify that the `Accept` trait and the `tokio_rustls::TlsStream` peer-certificate accessor still behave as assumed.

## Alternatives considered

**Pass the verified `EndpointId` from the verifier to the middleware via a process-wide side channel** (a `DashMap` keyed by connection ID, or `tokio::task_local!`, or a similar shared store written during `verify_client_cert` and read in the middleware). Rejected because it couples the verifier and the middleware through implicit state with a hard-to-reason-about lifetime. The verifier runs on rustls's internal scheduling, the middleware runs on tokio's request scheduling, and the connection between the two would have to be reconstructed by some kind of identifier — either a connection ID that both sides agree on (which neither side currently has) or a thread-local that survives across async boundaries (which is fragile). Worse, the side channel grows over time and needs cleanup logic that is easy to get wrong; a leaked entry is a slow memory growth bug, and a prematurely cleaned entry is a 500 that looks like a verifier failure but isn't. The acceptor approach avoids the side channel entirely by carrying the value along the same lifetime as the connection it belongs to.

**Roll back ADR-0016 and return to trusting `endpoint_id` from the request body, with an optional cross-check against the certificate CN.** Rejected because ADR-0016's reasoning has not changed. The body-as-identity model conflates "who is talking" with "what they are saying," and the cross-check version of it institutionalizes the wrong mental model for a defense that catches nothing in normal operation. The honest version of this alternative is that the implementation cost of the acceptor bridge is real but bounded — a few hundred lines of code, well-scoped, testable — and that cost does not justify giving up the architectural cleanliness ADR-0016 establishes. If, after attempting implementation, the acceptor approach turns out to be substantially harder than estimated, that is a reason to revisit this ADR honestly with a superseding ADR, not a reason to retreat to the body-trust model through the back door.

**Wait for upstream `axum-server` or `axum` to add native peer-certificate support.** Rejected because the upstream issues are years old with no active movement, Ken needs the bridge to land in Phase 1, and the work to wait would not be free anyway — the existing `endpoint_id.rs` placeholder and the comment that contradicts ADR-0016 cannot stay in the repository as a long-term answer. Building the bridge ourselves now also means Ken is not blocked on a third party for one of its load-bearing trust mechanisms, which is in keeping with the project's general posture of owning its critical paths.

## Notes

This ADR concretizes the propagation mechanism that ADR-0008 sketched in one sentence. It does not supersede ADR-0008 — the verifier design, the two-listener split, the database-backed enrollment check, and the rest of ADR-0008's decisions remain in force unchanged. This ADR fills in the one paragraph of ADR-0008 that turned out to assume an `axum-server` API surface that does not exist.

The crate-level skill at `.claude/skills/mtls-with-rustls/SKILL.md` should gain a short section pointing at the verifier/acceptor seam once this ADR is accepted. That update is part of the implementation work for issue #14, not a separate task.
